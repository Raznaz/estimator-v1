# /commit (локальный коммит)

Доведи текущую работу от грязного рабочего дерева до **локального коммита** на короткоживущей ветке, по правилам GitHub Flow и Conventional Commits. В `main` напрямую не коммитим.

Этот скилл покрывает только локальную часть: **ветка → коммит**. Push в `origin` и открытие PR — зона связанного скилла [`/pr`](../pr/SKILL.md). Запусти `/pr` после `/commit`, когда будешь готов оформить Pull Request.

## Использование

```
/commit                 # выбор ветки → коммит (после подтверждения)
/commit <сообщение>      # использовать данный subject (всё равно нормализуется к конвенции)
```

## Workflow

Репозиторий следует **GitHub Flow**: `main` всегда деплоится, вся работа идёт на короткоживущих ветках от `main`.

### 1. Выбор ветки

Сначала посмотри текущую ветку (`git branch --show-current`) и **спроси пользователя** через `AskUserQuestion`, какую использовать:

- **Создать новую ветку от свежего `main`** — для новой, не связанной с текущей работой задачи. Сначала обнови `main` (`git fetch` / `git pull`), потом ответвись от него.
- **Использовать текущую ветку** — когда продолжаешь работу, уже начатую на feature-ветке.

Сделай этот выбор через `AskUserQuestion` **до** любых других действий.

- **Никогда не коммить напрямую в `main`.** Если пользователь выбрал «текущую», но текущая ветка — `main`, всё равно создай новую ветку.
- Имена новых веток повторяют конвенцию коммитов: `<type>/<short-kebab-description>`, опционально со scope — например `feat/frontend-home-screen`, `fix/auth-refresh-401`, `chore/bump-prisma`.
- **Одна задача — одна ветка.** Держи ветки сфокусированными и короткоживущими.

### 2. Коммит (после подтверждения)

1. Застейджи релевантные изменения (`git add ...`). **Не стейджи** `.env`, `.env.local`.
2. Сформируй сообщение по Conventional Commits (см. формат ниже).
3. **Покажи пользователю** список staged-файлов (`git status --short`) и предложенное сообщение коммита, и **дождись подтверждения**.
4. Только после подтверждения выполни `git commit`.

На этом скилл останавливается — изменения зафиксированы локально. Сообщи пользователю, что для push и PR нужно запустить [`/pr`](../pr/SKILL.md).

## Формат сообщения коммита

Коммиты следуют [Conventional Commits](https://www.conventionalcommits.org/): `<type>(<scope>): <subject>`.

- **Типы:** `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore`, `revert`.
- **Scope** (опционально) называет затронутую область, например `backend`, `frontend`, `auth`, `transactions`, `prisma`, `deps`.
- **Subject** — в повелительном наклонении, со строчной буквы, без точки в конце (например `add transactions summary endpoint`).
- Breaking changes: добавь `!` после типа/scope (`feat(api)!: ...`) и/или футер `BREAKING CHANGE:`.

Примеры: `feat(transactions): add summary endpoint`, `fix(auth): refresh token on 401`, `chore(deps): bump prisma to 6.x`.

## Notes

- Не добавляй `Co-Authored-By` или атрибуцию «Generated with Claude Code» в коммиты.
- Коммить только когда пользователь об этом просит.
- DO **NOT** stage or commit `.env`, `.env.local`.
- Скилл **не пушит** и **не открывает PR** — это делает [`/pr`](../pr/SKILL.md).
